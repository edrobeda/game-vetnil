## Fase 6 — Migração do banco (pré-requisito)

### Problema
`server/db/schema.sql` está desatualizado. O código referencia colunas que ainda não existem no banco real.

### Solução
Gerar `server/db/migration_v2.sql` com os `ALTER TABLE` necessários. O usuário executa o SQL manualmente no banco.

### Alterações necessárias

**`clientes`**
- Adicionar `telefone VARCHAR(20)`
- Tornar `email` e `perfil` nullable

**`premios`**
- Adicionar `quantidade INTEGER`
- Ampliar constraint de `chance`: `DROP CONSTRAINT`, recriar como `CHECK (chance BETWEEN 1 AND 6)`

**`quiz`**
- Adicionar `ultima_resposta TEXT`
- Tornar `terceira` e `quarta` nullable
- Ampliar constraint de `correta` para suportar até 5

**`partidas`**
- Adicionar `status VARCHAR(30) DEFAULT 'jogando'`
- Adicionar `codigo VARCHAR(20) UNIQUE`
- Adicionar `entregue_em TIMESTAMP`
- Adicionar `operador VARCHAR(255)`

Atualizar também `schema.sql` para refletir o estado atual.

## Fase 7 — Página `/entrega` standalone

### Problema
A funcionalidade de entrega existe na aba do Manager (exige login JWT). O ROADMAP pede uma página separada de acesso mais simples para o balcão de retirada.

### Solução
Nova rota `/entrega` no React Router. Proteção por senha simples via `sessionStorage` (sem JWT). Senha configurada em `ENTREGA_PASS` no `.env` do servidor.

A `/entrega` usa endpoints **próprios** (não os do Manager), com middleware de auth separado que valida um header `x-entrega-token` contra a senha do `.env`:
- `POST /api/entrega/auth` — recebe `{ senha }`, retorna `{ token }` (hash simples ou a própria senha criptografada)
- `GET /api/entrega/:codigo` — busca partida por código (mesma lógica do Manager, middleware próprio)
- `POST /api/entrega/:codigo/confirmar` — confirma entrega (mesma lógica do Manager, middleware próprio)

A aba Entrega no Manager permanece intacta com seu JWT.

### Componentes
- `src/pages/Entrega/index.jsx` — tela de login simples + tela de busca/confirmação
- `src/pages/Entrega/Entrega.module.css`
- `server/routes/entrega.js` — endpoints `/api/entrega/*` com middleware de senha simples

## Fase 8 — Busca multi-tipo no quiosque

### Problema
O quiosque (`/jogo`) aceita apenas CPF. O ROADMAP pede suporte a telefone e código.

### Solução
Input único com auto-detecção de tipo na `TelaApresentacao` do Jogo:

| Padrão detectado | Tipo | Ação |
|---|---|---|
| Inicia com `EVT-` (após trim) | Código | `GET /api/cliente/por-codigo/:codigo` |
| Dígitos limpos têm 11 e primeiro dígito ≥ `0` com máscara CPF aplicada | CPF | `POST /api/cliente/validar` (já existe) |
| Dígitos limpos têm 10 ou 11 e input contém `(` ou ` ` (máscara telefone) | Telefone | `POST /api/cliente/validar` com `{ telefone }` |

**Regra de precedência:** EVT- > máscara telefone > CPF. O input aplica máscara automática: ao digitar `(` ou `+` força máscara de telefone; ao digitar só dígitos aplica máscara CPF; ao digitar `E`/`e` força campo texto livre (para código).

### Mudanças necessárias
- Frontend: placeholder e máscara dinâmicos, auto-detecção no submit
- Backend: ampliar `POST /api/cliente/validar` para aceitar `telefone`; novo endpoint `GET /api/cliente/por-codigo/:codigo`

## Fase 9 — Email manual pelo Manager

### Decisão
Email disparado manualmente pelo operador no Manager. Configurações SMTP são placeholder por enquanto.

### Solução
- Botão "Enviar email" na tabela de Clientes — visível somente quando cliente tem `email` cadastrado E `partidas.status IN ('premio_disponivel', 'premio_entregue')`
- Novo arquivo `server/routes/email.js`
- Endpoint `POST /api/email/enviar/:clienteId`
- Template HTML inline: nome, prêmio, código único, QR Code em base64 (gerado no servidor via `qrcode` npm)
- Ativar rota no `server/index.js` (linha comentada)

### Configuração `.env` (placeholder)
```
SMTP_HOST=smtp.exemplo.com
SMTP_PORT=587
SMTP_USER=noreply@exemplo.com
SMTP_PASS=senha_placeholder
SMTP_FROM_NAME=Game Roleta
```

## Decisões registradas

| Decisão | Motivo |
|---|---|
| `/entrega` com senha simples (não JWT) | Operador do balcão não precisa acesso ao Manager completo |
| Email manual, não automático | Usuário não tem SMTP configurado ainda |
| SMTP placeholder no `.env` | Config real será adicionada depois |
| Auto-detecção de tipo no input do quiosque | Melhor UX — operador não precisa escolher o tipo |
| Endpoints próprios em `/api/entrega/*` com middleware de senha simples | Operador do balcão não tem JWT do Manager; autenticação separada evita acoplamento |
